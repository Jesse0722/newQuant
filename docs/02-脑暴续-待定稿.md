## 一、页面结构与导航

**待定**：前端整体布局与路由结构。

**草案**：

```
/                     → 仪表盘（可选，MVP 可省略）
/data/stocks          → 股票列表
/data/sync            → 数据同步
/select               → 选股（配置条件 + 运行 + 结果列表）
/pools                → 股票池列表
/pools/:id            → 池内股票 + 快照 + 标签/记录
/backtest             → 回测（选标的 + 策略 + 参数 → 提交）
/backtest/:id         → 回测结果（曲线 + 指标 + 交易明细）
```

**待拍板**：
1. MVP 是否需要仪表盘？若需要，展示哪些汇总信息（如池子数量、最近选股/回测记录）？
2. 侧边栏导航是否按「数据 / 选股 / 股票池 / 回测」四块划分？有无其他分组方式？

## 二、因子模板具体列表

**待定**：预置因子模板的完整列表及参数定义。

**草案**（基于常见量化因子）：

| 模板 ID | 名称 | 表达式/逻辑 | 参数 |
|---------|------|-------------|------|
| momentum_5d | 5日动量 | (close - close.shift(5)) / close.shift(5) | - |
| momentum_20d | 20日动量 | 同上 | - |
| reversal_5d | 5日反转 | -1 * momentum_5d | - |
| volatility_20d | 20日波动率 | close.pct_change().rolling(20).std() | - |
| ma_cross | 均线乖离 | (close - ma(n)) / ma(n) | n |
| volume_ratio | 量比 | vol / vol.rolling(5).mean() | - |
| rsi | RSI | 标准 RSI 公式 | period |

**待拍板**：
1. MVP 先上哪些模板？建议至少：动量、反转、波动率、均线乖离、量比。
2. 因子模板是否支持「用户保存自定义因子」到库中（FactorDef 表），供后续选股复用？

## 三、选股条件筛选的 UI 交互

**待定**：用户如何配置选股条件（多条件 + 与/或 + 因子排序）。

**方案 A**：表单式。每一行一个条件：字段 + 运算符 + 值/引用；底部选择逻辑（与/或）；因子排序单独一块（选因子 + 取 TopN）。

**方案 B**：类 Query Builder。拖拽条件块，用连线表示逻辑关系。

**方案 C**：混合。简单条件用表单，复杂逻辑支持 DSL 输入（进阶用户）。

**待拍板**：
1. MVP 用方案 A 是否足够？方案 B 是否过度设计？
2. 选股「范围」如何配置：全市场 / 指定股票池 / 指定行业 / 排除 ST 等？

---

## 四、自定义策略的 UI 交互

**待定**：用户如何配置自定义买卖规则（表单配置 vs DSL）。

**表单配置**：买条件、卖条件各一组，每条件：field + op + (ref+params | value)；logic 选 and/or；position 选固定比例或等权。

**DSL**：如 `BUY: close > ma(20) and rsi < 30`，`SELL: close < ma(10)`。

**待拍板**：
1. MVP 是否只做表单配置？DSL 是否放到 V2？
2. 表单中「条件原子」的 field 下拉列表：close, high, low, open, vol, ma(n), rsi, macd 等，MVP 支持哪些？
3. 止损止盈是否进 MVP？若进，如何配置（固定比例 / ATR 倍数）？

---

## 五、回测任务队列与异步执行

**待定**：回测是同步阻塞还是异步任务。

**同步**：POST /api/backtest/run 直接执行，完成后返回结果。实现简单，但长时间回测会超时。

**异步**：提交任务返回 task_id，轮询 GET /api/backtest/{id}/status 或 WebSocket 推送进度；完成后 GET /api/backtest/{id}/result。

**待拍板**：
1. MVP 用同步是否可接受？若回测标的少（如单股、小池子）、区间短，同步通常几秒内完成。
2. 若选异步，后端用 Celery/Redis 还是内存队列 + 后台线程？MVP 复杂度会显著增加。

## 六、数据同步的限频与体验

**待定**：Tushare Pro 有调用频率限制，全市场日线同步可能很慢。

**策略**：
- 按股票逐个拉取，每请求后 sleep 一小段时间；
- 支持「指定股票列表」同步，不全量；
- 首次使用引导用户先同步 stock_basic，再按需同步部分标的的日线。

**待拍板**：
1. 是否在 UI 上明确提示「全市场同步较慢，建议先选标的」？
2. 是否支持「增量同步」（只拉本地缺失的日期）？实现上可查 daily_quote 表已有日期，只请求缺失区间。

## 七、错误处理与日志

**待定**：API 错误码规范、前端错误展示、后端日志级别。

**草案**：
- 统一错误响应：`{ code, message, detail? }`；
- 常见错误：TUSHARE_TOKEN 未配置、数据拉取失败、回测参数非法等；
- 日志：INFO 记录关键操作（同步、选股、回测），ERROR 记录异常堆栈。

**待拍板**：
1. 是否需要更细的错误码枚举（如 1001=token 缺失）？MVP 用简单 message 是否足够？
2. 日志是否落盘？若落盘，路径与轮转策略？

---

## 八、回测引擎选型

**待定**：backtrader vs 自写事件驱动。

| 方案 | 优点 | 缺点 |
|------|------|------|
| backtrader | 成熟、指标多、文档全 | 学习曲线、定制策略需继承类 |
| 自写 | 完全可控、易对接表单/DSL | 开发量大、易出 bug |

**待拍板**：
1. MVP 倾向 backtrader 还是自写？若自写，事件驱动的最小闭环（日线 bar → 信号 → 下单 → 绩效计算）工作量评估？

---

## 九、基准与组合回测

**待定**：收益曲线是否叠加基准（如沪深300）；组合回测的权重配置。

**基准**：需额外拉取指数日线（如 000300.SH），在回测结果中计算基准收益并叠加展示。

**组合权重**：等权 vs 自定义权重（每标的一个比例）。自定义权重需在回测请求中传 `{ ts_code: weight }`。

**待拍板**：
1. MVP 是否支持基准叠加？若支持，基准列表是否可配置（至少沪深300）？
2. 组合回测 MVP 是否只做等权？自定义权重是否 V2？

## 十、导出与报告

**待定**：导出 CSV 的范围；是否支持 PDF/Word 报告。

**CSV 导出**：选股结果、池内股票、交易明细、回测净值序列等，按需导出。

**报告**：自动生成 PDF/Word，包含曲线图、指标表、交易明细。需引入报告模板与渲染库（如 WeasyPrint、python-docx）。

**待拍板**：
1. MVP 的「导出 CSV」具体支持哪些？至少：选股结果、交易明细、净值序列。
2. 自动生成 PDF/Word 报告是否进 MVP？若进，模板风格与章节结构？
